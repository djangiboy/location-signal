# Wiom - Gate 0 Submission Template (Explained Version)

## What this document is

This is not a form.

This is a **thinking contract** that ensures every project:

- has a clear problem
- produces a measurable signal
- leads to learning
- connects to company value (NUT)

If you cannot fill a section, the project is not ready.

## What is a "signal" (read this first)

A **signal** is the earliest measurable change that tells you whether your intervention is working.

- It is NOT the final outcome (e.g., retention)
- It is NOT a vague indicator (e.g., "better experience")

It is: > A fast, measurable proxy that changes because of your action

### Example

Problem: Users drop off during onboarding

❌ Not a signal: "Retention improves"

✅ Signal: "% users completing onboarding step 3"

Why?

- observable quickly
- directly affected by onboarding changes
- tells you if your intervention worked

# SECTION A - Problem Definition

## 1\. Observed / Expected / Evidence

### What this means

You must define the problem in reality, not opinion.

- **Observed** = what is happening today
- **Expected** = what should happen
- **Evidence** = proof that there is a gap

### Example

Observed: 35% of users drop before completing onboarding Expected: At least 60% should complete onboarding Evidence: Funnel data from last 2 weeks shows drop at step 3

# SECTION B - Project Definition

## 2\. Objective

### What this means

What measurable change will happen if this works?

### Example

Increase onboarding completion rate from 35% → 50%

## 3\. Classification + Rationale

### What this means

Choose one type of work:

- Hygiene = fix something broken
- Efficiency = improve existing system
- Outcome Shift = change system behavior
- Capability Build = build system ability

Explain why it is NOT another type.

### Example

Classification: Efficiency Rationale: Improves onboarding flow without changing product model (not Outcome Shift)

## 4\. Root Cause

### What this means

Why is this happening?

Must be structural, not superficial.

### Example

Users drop because step 3 requires manual data entry, causing friction

## 5\. Leverage Level

### What this means

How deep is your intervention?

- Low = parameter change
- High = system change

### Example

LP2 (parameter change - simplifying form)

# SECTION C - Measurement

## 6\. Measurement Stack

### What this means

You must define all layers of measurement:

| Layer | Meaning                 |
| ----- | ----------------------- |
| L5    | Final outcome           |
| L4    | Driver                  |
| L3    | Signal (most important) |
| L2    | Execution proof         |
| L1    | System correctness      |

### Example

L5: Retention L4: Activation rate L3: Onboarding completion L2: % users reaching onboarding L1: System working without errors

## 7\. Tracking Table

### What this means

Define how you will measure success.

### Example

| Metric          | Baseline | Target | Frequency |
| --------------- | -------- | ------ | --------- |
| Completion rate | 35%      | 50%    | Daily     |

# SECTION D - Leading Signal (Critical)

## 8\. Leading Signal

### What this means

Your main measurement.

### Example

"% users completing onboarding step 3"

## 9\. Signal Validity

You must prove 3 things:

### Observability

Can we measure it quickly?

Example: "Users complete onboarding in real-time, so measurable within 1 day"

### Causality

Does it reflect the outcome?

Example: "Completion leads to activation, which leads to retention"

### Sensitivity

Will it change because of your action?

Example: "Reducing form fields directly affects completion rate"

## 10\. Signal Timing

### What this means

When will you see the signal?

### Example

Within 3 days of deployment

## 11\. Scope Constraint

### What this means

Can you observe the signal within your timeline?

### Example

Yes - onboarding completion happens immediately after change

# SECTION E - Ownership & Mapping

## 12\. Owner

### What this means

One person owns this project.

Not a team.

## 13\. Driver Mapping

### What this means

Which company driver does this affect?

### Example

Activation rate

## 14\. NUT Chain

### What this means

How does your project impact company value?

Format:

Project → Driver → NUT

### Example

Onboarding completion → Activation → Revenue per user

## 15\. Validation Path

### What this means

How does your signal prove this chain?

### Example

If completion increases, activation should increase

# SECTION F - Execution

## 16\. Time Class

### What this means

How long will this take?

### Example

Efficiency → 1 sprint

## 17\. Execution Plan

### What this means

How will this be executed?

- What agents do
- What humans do

### Example

Agent: generate UI variants Human: approve final design

# SECTION G - Learning Logic

## 18\. Hypothesis

### What this means

What do you expect to happen?

### Example

If we reduce onboarding steps, completion rate will increase by 10% within 5 days

## 19\. Risk Check

### What this means

What could go wrong?

### Example

Users may still drop due to external factors

## 20\. Learning Path

### What this means

What decision will you take based on signal?

### Example

If completion increases → scale change If not → redesign onboarding

# FINAL RULE

If you cannot define:

- a clear problem
- a measurable signal
- a testable hypothesis

Then: > This is not a project yet.

# END
